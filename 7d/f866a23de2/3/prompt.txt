### Gaps and Deviations Found

**Gap 1 — `threshold_reason_str` inspects the wrong counter when both thresholds trip simultaneously**

The spec (R6) says: "If both trip simultaneously (e.g., 3 consecutive blocks that also push the total to 10), use `"block_limit_consecutive"` (it fired first)."

The implementation checks consecutive first in `run_pre_dispatch` (lines 274-277 before 278-281), so `ThresholdTripped` is returned after the consecutive check trips. However, `threshold_reason_str` (lines 325-331) looks at `consecutive_block_count` at the moment `threshold_reason` is captured. The problem is that `threshold_reason` is captured *after the break from the guard loop*, so it reads the counter values at that point — which is correct. But there is a subtle structural issue: `threshold_reason_str` takes `consecutive_block_count` by value (not a flag indicating *which* counter tripped). If only the total counter trips (consecutive is, say, 2 out of 3) the function correctly returns `"block_limit_total"`. If consecutive trips first, consecutive is 3, function returns `"block_limit_consecutive"`. The logic is actually correct in the single-cause case.

However, the "simultaneous" edge case described in the spec (consecutive hits 3 AND total also hits 10 on the same block): `run_pre_dispatch` checks consecutive first, returns `ThresholdTripped` before checking total. So at the call site, `consecutive_block_count` is 3 and `threshold_reason_str(3)` returns `"block_limit_consecutive"`. This is correct. No actual gap here — the implementation is fine for this edge case.

**Gap 2 — Sequential path: partial `tool_results` from tools dispatched before the threshold tripped are silently discarded without explicitly discharging the conversation**

The spec says (R9 sequential path pseudocode and the architecture diagram):

```
if threshold_tripped:
  conversation.pop() + break (reason: block_limit_*)
  (tool_results discarded — batch abandoned, no message sent)
```

The implementation does match this — when `threshold_tripped`, it does `conversation.pop(); break` and the partially-collected `tool_results` vector is dropped. The conversation is popped back to before the assistant message. This is correct.

However, the sequential path `tool_results` vector may already contain results from tools that executed *before* the blocking tool hit the threshold. The spec says these are discarded ("batch abandoned, no message sent") and the implementation does discard them. This is correct. No gap.

**Gap 3 (Real) — The `threshold_reason_str` function does NOT capture which counter tripped at `ThresholdTripped` return time; it re-reads `consecutive_block_count` at the call site after the `break`**

Looking more carefully: `run_pre_dispatch` returns `ThresholdTripped` without encoding *which* counter tripped. The caller then calls `threshold_reason_str(consecutive_block_count)` — but by this point `consecutive_block_count` has already been incremented (inside `run_pre_dispatch`). Since `consecutive_block_count` is passed by `&mut usize` and mutated before returning `ThresholdTripped`, the count at the call site is the post-increment value. So if consecutive was 2 before the block, it becomes 3, and `threshold_reason_str(3)` correctly returns `"block_limit_consecutive"`. If total tripped first (consecutive was 1, total hit 10), `threshold_reason_str(1)` returns `"block_limit_total"`. This is correct per the logic. No actual gap.

`run_stop` filters hooks only by `h.event == "Stop"` — it does not apply `matches_tool`. The spec (R2) says `match_tool` is optional and when omitted fires for all tools. For Stop hooks there is no "current tool" concept, so `match_tool` on a Stop hook is meaningless. This is not a gap — the spec doesn't say Stop hooks should be filtered by tool. The implementation correctly collects all Stop-event hooks.
