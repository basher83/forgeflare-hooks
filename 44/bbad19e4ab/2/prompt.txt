- `HttpError`: 429, 503, 529, and unknown >= 500 as Transient; everything else as Permanent.
- `StreamTransient(_)`: Transient.
- `StreamParse(_)`: Permanent.
- `Api(e)`: Transient when `e.is_timeout() || e.is_connect()`, Permanent otherwise.
- `Json(_)`: Permanent.

The SSE `error` event handling correctly inspects `parsed["error"]["type"]` (the nested field, not the top-level `type`). `overloaded_error`, `api_error`, and `rate_limit_error` produce `StreamTransient`. `invalid_request_error` produces `StreamParse`.

In `api.rs` lines 339-344, the `_` wildcard arm routes unknown/absent types to `AgentError::StreamParse`, which is permanent. This directly contradicts the spec requirement. Unknown error type from the server should be `StreamTransient`, not `StreamParse`.

**R2. Retry with Exponential Backoff** — FULLY IMPLEMENTED.

The retry loop is `for attempt in 0..=MAX_RETRIES` (5 total iterations: 0 through 4 inclusive, giving 1 initial call + 4 retries). On transient error, when `attempt < MAX_RETRIES`, it computes delay and sleeps. `retry_after` from `HttpError` is used when present, capped at 60 — only for `HttpError` variants (correct). A value of 0 falls through without special treatment but `tokio::time::sleep(Duration::from_secs(0))` is effectively immediate, which matches the spec's intent.

**R4. Logging** — PARTIALLY IMPLEMENTED — minor deviation from spec format.

The spec says the log format should be: `[retry] Attempt {n}/4: {status} — waiting {delay}s`

The spec says `{status}` but the intent is the error description, and using the full `Display` is reasonable. This is a minor deviation in the format spec, not a functional problem.
