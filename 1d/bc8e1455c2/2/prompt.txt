### R4. GitHub Release

**Status: Two gaps found.**

The release step uses `gh release create "${TAG}" --title "${TAG}" --generate-notes` and attaches both tarballs. Title and auto-generated notes are correct.

Gap 1 — `--latest` flag is absent. The `gh release create` command does not include `--latest`. By default GitHub marks the release as latest only if it is the most recent by version sort, but the spec explicitly requires "Mark as latest release." Omitting `--latest` means this is implicitly dependent on GitHub's ordering logic rather than being explicitly enforced. It should be `gh release create "${TAG}" --latest ...`.

Gap 2 — Tarball naming uses `${GITHUB_REF_NAME}` which is set by GitHub for the tag ref, so functionally correct, but worth noting it relies on the environment variable rather than a more explicit extraction. This is not a spec deviation.

### R5. CI Gate

**Status: Not implemented as specified.**

This is the most significant gap. The spec states:

> "Release job depends on existing CI jobs (lint, audit, test, build) passing"
> "The release workflow either triggers CI as a prerequisite or the release workflow file includes the CI validation steps inline per matrix leg"

What was implemented: the `build` job in `release.yml` runs `cargo fmt --check`, `cargo clippy`, `cargo audit`, and `cargo test` inline on each matrix leg before building. So validation does happen.

However, the spec's primary intent (confirmed by the architecture diagram showing a separate "CI gate" box that "must pass" before the build matrix runs) was to use `ci.yml`'s jobs as a prerequisite. The `ci.yml` file has `workflow_call:` as a trigger, which means it was designed to be called from other workflows. The `release.yml` never calls or depends on `ci.yml` at all.

The practical consequence: CI runs independently from the release workflow. A tag push triggers both `ci.yml` and `release.yml` simultaneously and independently. The release build does not wait for the separate CI workflow to pass — it re-runs equivalent steps inline. If CI fails on the tag push, the release workflow may still succeed and publish artifacts.

The spec's architecture diagram shows `CI gate → build matrix → release`, a sequential dependency. The implementation runs them in parallel with no cross-workflow dependency.

This is a structural deviation. The `workflow_call` support in `ci.yml` exists precisely to enable the intended pattern:

```yaml
jobs:
  ci:
    uses: ./.github/workflows/ci.yml
  build:
    needs: [ci]
    ...
```

That wiring is absent from `release.yml`.

Deviation — the pinned SHAs for shared actions differ between `ci.yml` and `release.yml`:

- `actions/checkout`: `ci.yml` uses `de0fac2e4500dabe0009e67214ff5f5447ce83dd` (tagged as v6 in comments), `release.yml` uses `34e114876b0b11c390a56381ad16ebd13914f8d5` (tagged as v4.3.1). These are different commits pointing to different versions of the same action.

The spec says "Pin all action SHAs (same convention as `ci.yml`)." Using different SHA pins for `actions/checkout` violates this consistency requirement.
